\begin{abstract}
Public transit agencies publish large General Transit Feed Specification (GTFS) datasets that encode routes, trips, calendars, and stop-level schedules for complex multi-modal systems. Although the format is standardized, large-scale analysis is still cumbersome because even basic questions require joining several files, normalizing large numbers of records, and aggregating patterns across cities. This project builds a parallel pipeline for multi-city transit schedule analysis using MPI, MPI-I/O, and CUDA on the AiMOS supercomputer. The pipeline downloads current and archive-backed GTFS feeds from several northeastern and midwestern providers, converts them into a compact fixed-width binary format, and reads them collectively with \texttt{MPI\_File\_read\_at\_all}. MPI ranks exchange boundary records so headway calculations remain correct across partitions, then reduce provider-level statistics globally. Within each rank, either a CPU-only MPI backend or a CUDA backend computes hourly departures, trip-duration summaries, and headway-related metrics. The resulting dataset contains 3,703,385 trip records across MTA, CTA, MBTA, NJ Transit, and SEPTA. On AiMOS, the CPU path shows useful strong scaling from 50.6\,ms at two ranks to 20.4\,ms at sixteen ranks on the full dataset and reasonably stable weak-scaling behavior as total work increases from 300,000 to 2.4 million records. The GPU kernel scales locally, but end-to-end runtime remains dominated by fixed CUDA overhead, so the hybrid MPI+CUDA path is slower than CPU-only MPI for this workload.
\end{abstract}

\section{Introduction}
Public transit systems publish schedule data at a scale that is awkward for interactive analysis and expensive to normalize repeatedly. A single GTFS feed is a relational bundle of routes, trips, stop times, stops, calendars, and metadata. In practice, a multi-city study means integrating many agency feeds, handling format quirks, deriving route-level and trip-level features, and then aggregating them into system-level statistics. This fits HPC well: the input is large, preprocessing is expensive, and the final analysis needs both local computation and global coordination.

This project targets that problem with a parallel system built around three steps. First, we convert multi-file GTFS schedules into a fixed-width binary trip-record dataset that supports collective file access. Second, we use MPI and MPI-I/O to distribute that dataset across ranks, exchange boundary state, and reduce global aggregates. Third, we accelerate local analytics with CUDA so that each rank can use the GPU for bucketized schedule statistics and headway accumulation. The resulting workflow is not embarrassingly parallel because route headways cross partition boundaries, city-level statistics require collective reduction, and the input itself is read from a shared parallel file using MPI-I/O.

The application focus is \emph{scheduled service analysis}, not ridership analysis. Because the current pipeline uses static GTFS schedules rather than fare-card, passenger-count, or GTFS-Realtime delay data, the findings in this report concern service intensity, service mix, trip duration, and scheduled headway regularity. Those are still useful operational indicators, and they produce a workload with real communication and reduction costs.

The main contributions of this project are:
\begin{itemize}[leftmargin=*]
  \item a GTFS preprocessing pipeline that converts heterogeneous public feeds into one compact binary dataset suitable for collective MPI-I/O;
  \item a distributed analytics engine that combines MPI partitioning, boundary exchange, reduction, and CUDA local processing;
  \item a provider-level transit analysis across five major agencies using real GTFS schedule data; and
  \item an experimental design for strong scaling, weak scaling, and CPU-versus-GPU comparison on AiMOS.
\end{itemize}

\section{Related Work}
GTFS has become a standard substrate for transit network analysis. The official GTFS schedule and GTFS-Realtime specifications define the semantics and field structure for static and live transit data~\cite{gtfs_schedule_ref,gtfs_realtime_ref,gtfs_overview}. Those specifications make ecosystem-wide tooling possible, but they do not remove the underlying cost of integrating schedule, calendar, and stop-time files into analytical features.

Several prior works show the analytical value of GTFS. Fayyaz et al.~\cite{fayyaz2017gtfs} developed an efficient GTFS-enabled algorithm for dynamic transit accessibility analysis. Fortin et al.~\cite{fortin2016gtfs} used GTFS data to build graph-oriented transit network indicators, while Guthrie et al.~\cite{guthrie2017accessibility} used GTFS for accessibility scenario analysis in sketch planning. Pereira et al.~\cite{pereira2021r5r} introduced a multimodal routing framework that relies on GTFS-style network data and supports accessibility-oriented queries at scale.

More recent work uses GTFS in interactive and operational settings. Devunuri and Lehe~\cite{devunuri2025transitgpt} showed that GTFS datasets can support flexible analytical querying through large language model interfaces. Javanmard et al.~\cite{javanmard2025realtime} and Zhou et al.~\cite{zhou2024latency} use GTFS-Realtime for uncertainty-aware accessibility and operational decision support. Together, these studies point to the need for scalable infrastructure that can process multiple feeds repeatedly.

On the HPC side, we rely on standard distributed and accelerator programming models. The MPI 4.1 standard provides the collective communication and MPI-I/O interfaces used by our code~\cite{mpi41}. Thakur et al.~\cite{thakur1999romio} describe the collective-I/O mechanisms in ROMIO that motivate our use of \texttt{MPI\_File\_read\_at\_all} over a shared binary dataset. For GPU acceleration, the implementation follows the CUDA programming model described by Nickolls et al.~\cite{nickolls2008cuda} and NVIDIA’s programming guide~\cite{cuda_guide}. Our contribution is to apply these standard HPC tools to a transit-schedule workload with shared input, boundary-dependent headways, and provider-level aggregation.

\subsection{Trip-Record Representation}
The preprocessing stage converts each GTFS trip into a fixed-width record:
\begin{quote}
\small\ttfamily
(city\_id, feed\_id, route\_type, route\_hash, service\_mask,\\
direction\_id, start\_secs, end\_secs, duration\_secs, stop\_count,\\
trip\_hash, reserved)
\end{quote}
This representation is intentionally compact. It avoids expensive string parsing during parallel execution and allows the MPI layer to partition input by record count rather than by text-file boundaries.

Records are sorted by:
\begin{quote}
\small\ttfamily
(city\_id, route\_hash, direction\_id, start\_secs, trip\_hash)
\end{quote}
That sort order is critical because the route-headway computation depends on adjacent records within each city, route, and direction.

\subsection{Analytical Questions}
This report asks two sets of questions:
\begin{enumerate}[leftmargin=*]
  \item \textbf{HPC questions}: How does the hybrid MPI+CUDA implementation behave under strong scaling, weak scaling, and CPU-versus-GPU comparison? How much time is spent on file I/O, communication, reduction, and compute?
  \item \textbf{Transit questions}: How do providers differ in peak/off-peak concentration, scheduled headway structure, hourly service profile, and mode composition?
\end{enumerate}

These two question sets are linked. Faster analysis makes it easier to compare providers across larger datasets and finer schedule detail.

\subsection{Preprocessing}
The preprocessing stage is implemented in Python. It reads \texttt{routes.txt}, \texttt{trips.txt}, \texttt{stop\_times.txt}, and calendar metadata and then derives one record per trip. During implementation, several real-world GTFS packaging differences had to be handled explicitly:
\begin{itemize}[leftmargin=*]
  \item some feeds include \texttt{calendar\_dates.txt} but no \texttt{calendar.txt};
  \item some providers package multiple nested GTFS archives inside one top-level zip; and
  \item route-type values are not always limited to the canonical small GTFS set, so extended values must later be normalized for reporting.
\end{itemize}
A preprocessing pipeline for real feeds has to handle those packaging differences.

\subsection{MPI-I/O and Partitioning}
At runtime, the analyzer opens the binary dataset collectively with \texttt{MPI\_File\_open}, obtains the file size, converts it to a record count, and partitions the record range across ranks. Each rank then issues a collective \texttt{MPI\_File\_read\_at\_all} to fetch its contiguous record block.

This avoids a serial rank-0 read followed by a broadcast or scatter. It also puts the input path inside the measured workload, which matters here because I/O is a real part of the runtime.

\subsection{Boundary Exchange}
Route headways cross partition boundaries. If one rank ends with trip $t_i$ and the next rank begins with trip $t_{i+1}$ on the same route-direction pair, the headway between them must still be counted. The implementation therefore uses \texttt{MPI\_Sendrecv} so each rank can compare its first local trip to the previous rank’s last trip.

\subsection{CPU and CUDA Backends}
The local analytics stage supports two interchangeable backends:
\begin{itemize}[leftmargin=*]
  \item \textbf{CPU backend}: MPI-distributed host-side accumulation for all statistics;
  \item \textbf{CUDA backend}: GPU-accelerated local analysis within each rank.
\end{itemize}

The CUDA path uses the rank’s node-local communicator to select a GPU deterministically and then copies the local record partition to device memory. A CUDA kernel computes departures per time bucket, route-type counts, trip-duration sums, stop-count sums, and headway-related aggregates.

\begin{algorithm}[t]
\caption{Distributed Transit Analytics}
\label{alg:analysis}
\begin{algorithmic}[1]
\Require Sorted binary trip-record dataset; backend $\in \{\texttt{cpu}, \texttt{cuda}\}$
\State Open dataset collectively with MPI-I/O
\State Partition records across ranks
\State Read local block with \texttt{MPI\_File\_read\_at\_all}
\State Exchange one boundary record with neighboring rank
\If{backend = \texttt{cuda}}
  \State Copy local block to GPU
  \State Launch CUDA kernel to accumulate local statistics
\Else
  \State Accumulate local statistics on CPU
\EndIf
\State Reduce local arrays and sums to rank 0
\State Emit timing, provider summaries, and optional CSV reports
\end{algorithmic}
\end{algorithm}

\subsection{Global Reduction}
Each rank computes local counts, sums, and squared sums. Rank~0 receives the global totals with \texttt{MPI\_Reduce}, then derives:
\begin{itemize}[leftmargin=*]
  \item hourly departure profiles,
  \item peak/off-peak ratios,
  \item route-type counts,
  \item mean and coefficient of variation of headways,
  \item bunching and wide-gap rates, and
  \item trip-duration summaries.
\end{itemize}
Because the local arrays must be combined to produce provider-level outputs, the analysis necessarily includes a collective aggregation stage.

\section{Experimental Methodology}
All cluster experiments target AiMOS POWER9 nodes with NVIDIA V100 GPUs and use the module stack
\begin{quote}
\texttt{module load xl\_r spectrum-mpi cuda}
\end{quote}
as described in the course materials. The build process compiles the MPI logic with \texttt{mpicc} and the CUDA logic with \texttt{nvcc}, then links the resulting objects into a single executable.

The performance study follows the assignment requirements:
\begin{enumerate}[leftmargin=*]
  \item \textbf{Strong scaling}: fix the total record count and increase the rank count.
  \item \textbf{Weak scaling}: fix the records per rank and grow the total dataset proportionally.
  \item \textbf{CPU-only MPI vs.\ hybrid MPI+CUDA}: compare the two backends directly.
\end{enumerate}

Each run reports:
\begin{itemize}[leftmargin=*]
  \item MPI-I/O time,
  \item boundary-exchange time,
  \item local compute time,
  \item reduction time, and
  \item total runtime.
\end{itemize}
We report this breakdown so we can separate bottlenecks in shared-file access, communication, and computation.

\section{Validation and Limitations}
\subsection{Validation Strategy}
Because this project combines preprocessing, distributed execution, and GPU acceleration, correctness had to be checked at more than one stage. The first level of validation was at preprocessing time. For each provider group, the generated binary dataset was compared against simple feed-level expectations such as the number of routes retained, the number of trip records emitted, and whether basic schedule summaries were plausible. This step was especially important for feeds that did not follow the most common GTFS packaging pattern. Missing \texttt{calendar.txt} files, nested zip archives, and extended route-type values could all introduce silent errors if they were not handled explicitly.

The second level of validation was at the analytics stage. The provider-level results reported by the analyzer were checked for consistency across repeated runs and across backend choices. In particular, the CPU-only MPI backend and the hybrid MPI+CUDA backend were expected to produce the same aggregate outputs even when their runtime costs differed substantially. Matching provider summaries across those two paths served as a practical correctness check for the CUDA kernel, the device-memory transfers, and the final reduction logic. Distributed runs were also checked across different MPI rank counts to make sure that partitioning the dataset did not change the final totals.

Boundary conditions required separate attention because route headways depend on adjacent records that may fall on different ranks. Without an explicit boundary exchange, a distributed run could undercount or distort headways at partition edges. To guard against that, the implementation exchanges one record of boundary state with neighboring ranks and then recomputes the first local headway using the previous rank's last record when appropriate. This is a small communication step, but it is necessary for correctness. Its effect is visible in the fact that headway-based provider summaries remain stable across rank counts rather than drifting as the partition shape changes.

\subsection{Current Limitations}
The main limitation of the current study is that it analyzes scheduled service rather than observed operations. GTFS static feeds describe planned trips, not whether vehicles actually ran on time, whether trips were canceled, or how passenger demand was distributed across the network. As a result, the headway and irregularity metrics in this report should be read as schedule-structure summaries, not as direct reliability measures. A provider may show wide scheduled gaps because it is designed around commuter or regional service, while a dense urban provider may show frequent short scheduled headways without necessarily delivering those headways in practice.

A second limitation is the way historical coverage is represented. To make the workload larger and more realistic, the preprocessing stage now incorporates archive-backed GTFS snapshots for several providers. That improves computational scale, but the current binary schema does not preserve an explicit snapshot timestamp or materialize trips by service date. In other words, the dataset is richer than a single-feed snapshot, but it still aggregates multiple schedule vintages into one trip-record collection. The provider findings should therefore be interpreted as schedule-structure summaries over the collected snapshots rather than as a single clean service period.

There are also modeling choices in the preprocessing stage that simplify the problem. The binary trip-record representation is intentionally compact and does not preserve every field that might be useful in later analyses. Likewise, route-type normalization is necessary to compare providers cleanly, but it compresses some of the variety present in extended GTFS route codes. Because the archive-backed dataset includes multiple schedule vintages, route counts in the provider-analysis section are best interpreted as snapshot-aware route keys rather than as a provider's present-day route map.

Finally, the GPU result remains limited by workload shape even after the dataset expansion. The local analytics kernel does benefit from GPU execution, but the full application remains dominated by fixed costs such as CUDA runtime initialization and host-device transfers. This means the present report captures an important hybrid-programming result, but not yet a case where the GPU wins end-to-end. Larger datasets, repeated analyses over resident device data, or heavier local kernels would likely change that balance. The current implementation therefore demonstrates a correct hybrid workflow more clearly than it demonstrates a production-ready acceleration point.

\section{Strong Scaling Parallel Performance}
\label{sec:strong-scaling}
The figures in this section were generated from the AiMOS output stored in \texttt{experiment\_results.csv}. The committed strong-scaling sweep completed for the CPU backend on the full archive-backed dataset, so the main question here is how much benefit MPI alone gets from additional ranks when the total workload is fixed at 3,703,385 records.

\begin{figure}[H]
  \centering
  \includegraphics[width=\columnwidth]{../results/plots/strong_scaling_total_time.png}
  \caption{Strong scaling of the CPU-only MPI backend on the full archive-backed dataset.}
  \label{fig:strongscaling}
\end{figure}

\begin{figure}[H]
  \centering
  \includegraphics[width=\columnwidth]{../results/plots/strong_cpu_speedup.png}
  \caption{CPU strong-scaling speedup relative to the two-rank run.}
  \label{fig:strongspeedup}
\end{figure}

\begin{figure}[H]
  \centering
  \includegraphics[width=\columnwidth]{../results/plots/strong_cpu_breakdown.png}
  \caption{Strong-scaling runtime breakdown for the CPU-only MPI backend.}

The strong-scaling study fixes the total record count at 3,703,385 records and increases the number of MPI ranks. The CPU-only MPI backend improves from 50.6\,ms at two ranks to 26.2\,ms at four ranks, 24.8\,ms at eight ranks, and 20.4\,ms at sixteen ranks. That is not ideal linear scaling, but it is still useful: the sixteen-rank run is about 2.48$\times$ faster than the two-rank run on the same full dataset.

Figures~\ref{fig:strongscaling}--\ref{fig:strongbreakdown} show why the gains flatten. The local compute phase drops sharply from 20.2\,ms at two ranks to 3.49\,ms at sixteen ranks, but the I/O component stays in the 11--28\,ms range and therefore becomes a larger fraction of the total runtime. Boundary exchange and reduction remain negligible. In other words, the CPU backend does benefit from more ranks, but the shared-file read and other fixed costs keep the scaling curve from being dramatic.

The weak-scaling study fixes the records per rank and increases the total workload with the number of MPI ranks. The CUDA backend behaves differently. Its weak-scaling runtime grows from 0.292\,s at two ranks to about 0.488\,s at sixteen ranks, and it remains far slower than the CPU baseline at every scale. The weak-scaling plots therefore reinforce the same architectural lesson as the summary run: the hybrid MPI+CUDA path is correct, but it is still overhead-bound for this workload.

\section{Analysis of Performance Results}
\Needspace{14\baselineskip}
The measured results make the dominant bottlenecks clearer than the original design-level expectations:
\begin{itemize}[leftmargin=*]
  \item \textbf{CUDA runtime overhead dominates total time.} The measured GPU compute phase is only a few milliseconds, but total application time is hundreds of milliseconds, implying that device setup and transfer overhead are the primary bottleneck.
  \item \textbf{MPI-I/O matters more after the dataset expansion.} On the full strong-scaling runs, I/O stays in the 11--28\,ms range and becomes a meaningful fraction of the total time even though it is still much smaller than the GPU end-to-end overhead.
  \item \textbf{Boundary exchange and reduction are negligible at current scale.} Both remain in the microsecond range, so they do not dominate either backend in the present dataset size.
  \item \textbf{The CPU baseline remains strong even on the larger dataset.} With 3,703,385 records total, the host-side loops are still inexpensive enough that GPU acceleration cannot amortize its fixed costs.
\end{itemize}

The main performance lesson is architectural. A hybrid MPI+CUDA design is not automatically faster than MPI on CPUs; it becomes faster only when the local compute intensity is high enough to amortize the GPU’s fixed runtime costs. In the current implementation, that threshold has not been reached, so the GPU path is slower end-to-end even though the local kernel scales in the expected direction.

\section{Discussion}
The provider-level analytics show why this application is worth parallelizing. After adding archive-backed schedule snapshots, the dataset now contains more than 3.7 million trip records across five systems. New York contributes about 1.09 million records, Boston about 1.69 million, and Philadelphia about 561,000. A single local pass is still manageable, but the larger dataset makes the experimental results more defensible and gives the shared-file and reduction stages enough work to show up clearly in the timing breakdown.

The provider findings also line up with the service patterns one would expect. NJ Transit is still the most peak-concentrated and the most headway-sparse among the providers studied here, which fits a commuter-oriented system. CTA and MTA have the shortest route-level median headways, consistent with dense urban service, while SEPTA sits between those urban systems and NJ Transit. MBTA remains the shortest-duration system on average and also shows the highest bunch-rate proxy in the current aggregated schedule dataset. These are schedule-based findings, not passenger-based conclusions, but they still give a useful cross-city picture.

One practical lesson from the data-engineering stage is that publicly available GTFS feeds are cleaner in schema than in packaging. Missing \texttt{calendar.txt} files, nested archives, and extended route-type codes all had to be handled explicitly before the HPC work could begin. That preprocessing burden is easy to skip over in an algorithm sketch, but it matters in a system built on real feeds.

Another lesson is that no single metric is enough. Peak/off-peak ratios alone make the agencies look fairly similar, but route intensity, service span, headway distributions, and irregularity proxies separate them more clearly. MTA and CTA are both high-intensity systems by trips per route, yet MTA keeps that intensity across a longer service-day span. MBTA has a much shorter average trip duration than the commuter-oriented providers, while NJ Transit's peak concentration is easier to interpret when paired with its longer durations, larger headways, lower route intensity, and much higher wide-gap rate.

\section{Conclusions and Future Work}
This project shows that GTFS schedule feeds can be turned into a parallel analytics workload using MPI, MPI-I/O, and CUDA. The binary trip-record representation supports collective shared-file access, the boundary-exchange step keeps headway calculations correct across partitions, and the CUDA path provides a direct comparison against CPU-only MPI. The provider-side results show clear differences in peak concentration, route headways, service span, trip duration, route intensity, and mode composition across MTA, CTA, MBTA, NJ Transit, and SEPTA, even after the dataset was expanded with archive-backed schedule snapshots.

The completed AiMOS timing study produced a clear result. On the 3,703,385-record full dataset, the CPU-only MPI backend improved from 50.6\,ms at two ranks to 20.4\,ms at sixteen ranks, and the weak-scaling runs stayed in a narrow 10.6--15.3\,ms band while total work increased from 300,000 to 2.4 million records. The MPI+CUDA path runs correctly and its local kernel time scales in the expected direction, but the full application is still slower than CPU-only MPI because GPU initialization and transfer overhead dominate the end-to-end runtime. In this version of the workload, the GPU helps the kernel and hurts the total runtime.

Beyond the current prototype, the most promising extensions are:
\begin{itemize}[leftmargin=*]
  \item integrating \texttt{calendar\_dates.txt} exceptions more explicitly,
  \item expanding \texttt{frequencies.txt} into generated trips when needed,
  \item adding GTFS-Realtime delay records to support true reliability analysis,
  \item experimenting with NVMe staging versus direct shared-file access, and
  \item extending the workload to more cities and longer time horizons.
\end{itemize}

With those additions, the project could grow from a schedule-analytics case study into a real-time transit-performance platform while still serving as a solid hybrid HPC workload.
